# Аудит безопасности

## Реализованные меры

- Административные операции защищены middleware `requireAuth` и `requireAdmin`.
- Токен авторизации передается через заголовок `Authorization: Bearer`.
- Входные данные задач, подразделений и пользователей проверяются на сервере.
- Новые пароли хешируются SHA-256 с индивидуальной солью.
- Старые SHA-1 хеши поддержаны только для совместимости с уже созданными данными.
- Включены HTTP-заголовки `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`.
- Резервная выгрузка не раскрывает хеши паролей.
- Ведется журнал активности и статистика запросов.

## Проверка

1. Выполнить вход администратором.
2. Запросить `GET /api/security-audit` с Bearer-токеном.
3. Проверить, что пользователь без роли администратора получает `403` на административных маршрутах.
4. Запустить автоматические тесты: `npm test`.

## Остаточные риски

- Для промышленной эксплуатации нужно заменить хранение в JSON на СУБД.
- Для публичного размещения нужно включить HTTPS на уровне хостинга.
- Для продакшена рекомендуется добавить ограничение частоты входа и полноценное управление сроком жизни токенов.
